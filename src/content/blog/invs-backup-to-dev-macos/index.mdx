## ส่วนที่ 1 — เตรียมเครื่อง macOS ให้พร้อม

### 1.1 ตรวจสอบ Docker

<NarrativeSection>

SQL Server ไม่มี native binary สำหรับ macOS (ทั้ง Intel และ Apple Silicon) วิธีมาตรฐานที่ Microsoft รองรับอย่างเป็นทางการคือใช้ Docker

</NarrativeSection>

ตรวจสอบว่า Docker ติดตั้งแล้วหรือยัง:

```bash
docker --version
```

ผลลัพธ์ที่ถูกต้องควรหน้าตาประมาณนี้:

```
Docker version 29.2.1, build abcdef1
```

ถ้าขึ้น `command not found` แสดงว่ายังไม่ได้ติดตั้ง Docker ให้ดาวน์โหลด Docker Desktop จาก [docker.com](https://www.docker.com/products/docker-desktop)

ติดตั้งเสร็จแล้วให้เปิด Docker Desktop รอจนไอคอนใน menu bar กลายเป็น "Running" แล้วค่อยทำขั้นตอนถัดไป

<SideNote type="note">
  <p><strong>หมายเหตุสำหรับ Apple Silicon (M1/M2/M3):</strong> Docker Desktop รุ่นใหม่รองรับ Rosetta 2 และสามารถรัน image ที่ build สำหรับ <code>linux/amd64</code> ได้ ซึ่งครอบคลุม SQL Server image อย่างเป็นทางการของ Microsoft</p>
</SideNote>

### 1.2 ตรวจสอบ Homebrew

Homebrew จำเป็นสำหรับติดตั้ง ODBC driver ในภายหลัง:

```bash
brew --version
```

ถ้ายังไม่มี:

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

## ส่วนที่ 2 — รัน SQL Server บน macOS ด้วย Docker

### 2.1 ทำความเข้าใจ Docker Image ของ SQL Server

<NarrativeSection>

Microsoft เผยแพร่ image อย่างเป็นทางการผ่าน Microsoft Container Registry (MCR) ไม่ใช่ Docker Hub ทั่วไป Image นี้รัน SQL Server บน Linux ซึ่งเป็นเหตุผลที่ทำงานได้บน Docker ทุกระบบปฏิบัติการ

</NarrativeSection>

```
mcr.microsoft.com/mssql/server:2022-latest
```

### 2.2 รัน Container

```bash
docker run -d \
  --name sqlserver \
  -e "ACCEPT_EULA=Y" \
  -e "SA_PASSWORD=StrongPassword123!" \
  -p 1433:1433 \
  mcr.microsoft.com/mssql/server:2022-latest
```

<CodeExplainer title="อธิบาย Docker run options">
  <details>
    <summary><code>-d</code></summary>
    <p>รันใน background (detached mode) ไม่ block terminal</p>
  </details>
  <details>
    <summary><code>--name sqlserver</code></summary>
    <p>ชื่อ container ที่เราจะอ้างถึงในคำสั่งอื่น</p>
  </details>
  <details>
    <summary><code>ACCEPT_EULA=Y</code></summary>
    <p>ยืนยันว่ายอมรับ End-User License Agreement ของ Microsoft SQL Server ถ้าไม่ใส่ container จะไม่ start</p>
  </details>
  <details>
    <summary><code>SA_PASSWORD=StrongPassword123!</code></summary>
    <p>รหัสผ่านของ user <code>sa</code> (System Administrator) ต้องมีความยาว 8+ ตัว มีตัวพิมพ์ใหญ่ ตัวพิมพ์เล็ก ตัวเลข และสัญลักษณ์ ถ้าไม่ครบ container จะ exit</p>
  </details>
  <details>
    <summary><code>-p 1433:1433</code></summary>
    <p>map port 1433 ของ container ออกมาที่ port 1433 ของเครื่อง host รูปแบบคือ host:container</p>
  </details>
</CodeExplainer>

### 2.3 ตรวจสอบว่า Container ทำงานอยู่

```bash
docker ps
```

ผลลัพธ์ที่ถูกต้อง:

```
CONTAINER ID   IMAGE                                        STATUS         PORTS                    NAMES
307fb4a0ebca   mcr.microsoft.com/mssql/server:2022-latest   Up 2 minutes   0.0.0.0:1433->1433/tcp   sqlserver
```

สิ่งสำคัญที่ต้องดู:
- **STATUS** ต้องเป็น `Up` ไม่ใช่ `Exited`
- **PORTS** ต้องเห็น `0.0.0.0:1433->1433/tcp`

<InfoBox title="ถ้า STATUS เป็น Exited" type="warning">
  <p>ให้ดู log ด้วยคำสั่ง <code>docker logs sqlserver</code></p>
  <p>error ที่พบบ่อยที่สุดคือ password ไม่ผ่านเกณฑ์ความปลอดภัย ให้ลบ container แล้วรันใหม่ด้วย password ที่แข็งแรงกว่า:</p>

```bash
docker rm sqlserver
# แล้วรัน docker run ... ใหม่
```

</InfoBox>

### 2.4 รอให้ SQL Server พร้อม

SQL Server ใช้เวลาประมาณ 10–30 วินาทีหลัง container start ก่อนที่จะรับ connection ได้ วิธีตรวจสอบ:

```bash
docker exec -it sqlserver /opt/mssql-tools/bin/sqlcmd \
  -S localhost -U sa -P "StrongPassword123!" -Q "SELECT 1"
```

ถ้าเห็นผลลัพธ์เป็น `1` แสดงว่า SQL Server พร้อมใช้งานแล้ว

## ส่วนที่ 3 — เตรียมไฟล์ Backup

### 3.1 โครงสร้างโฟลเดอร์

สร้างโฟลเดอร์สำหรับเก็บไฟล์ backup ในเครื่อง:

```bash
mkdir -p backup
```

วางไฟล์ backup ไว้ที่ `backup/INVS.bak` แล้วตรวจสอบ:

```bash
ls -lh backup/
```

### 3.2 Copy ไฟล์เข้า Container

<NarrativeSection>

SQL Server ที่รันใน container จะเข้าถึงไฟล์บนเครื่อง host โดยตรงไม่ได้ ต้อง copy เข้าไปก่อน

</NarrativeSection>

```bash
# สร้างโฟลเดอร์ backup ใน container ก่อน
docker exec sqlserver mkdir -p /var/opt/mssql/backup

# copy ไฟล์เข้า container
docker cp backup/INVS.bak sqlserver:/var/opt/mssql/backup/
```

ตรวจสอบว่า copy สำเร็จ:

```bash
docker exec sqlserver ls -lh /var/opt/mssql/backup/
```

<SideNote type="tip">
  <p><strong>ทำไมต้อง copy เข้า /var/opt/mssql/backup/?</strong> เป็น path มาตรฐานที่ SQL Server บน Linux มี permission เขียน/อ่านได้ ถ้าวางใน path อื่นอาจเจอปัญหา permission denied ตอน restore</p>
</SideNote>

### 4.2 ตรวจสอบ Logical Name ก่อน Restore

<NarrativeSection>

ก่อน restore ต้องรู้ "Logical Name" ของไฟล์ข้อมูลและ log ใน backup เพราะจะต้องใช้ตอนระบุ path ปลายทาง

</NarrativeSection>

```sql
RESTORE FILELISTONLY
FROM DISK = '/var/opt/mssql/backup/INVS.bak'
```

ผลลัพธ์ที่ได้จะคล้ายนี้:

```
LogicalName    PhysicalName                              Type
-----------    -----------------------------------------  ----
invs           C:\MSSQL\DATA\INVS.mdf                    D
invs_log       C:\MSSQL\DATA\INVS_log.ldf                L
```

`D` คือ data file (`.mdf`) และ `L` คือ log file (`.ldf`) — จด `LogicalName` ทั้งสองตัวไว้ จะใช้ใน `MOVE` clause ในขั้นตอนถัดไป

<SideNote type="note">
  <p><strong>ทำไมต้องดู LogicalName?</strong> Backup ถูกสร้างบน Windows Server ซึ่ง physical path เป็น <code>C:\MSSQL\...</code> แต่ container ของเราเป็น Linux ดังนั้น SQL Server จะไม่สามารถ restore ไปยัง path เดิมได้ ต้องใช้ <code>MOVE</code> เพื่อบอก path ใหม่</p>
</SideNote>

### 4.3 รัน Restore

แทนที่ `invs` และ `invs_log` ด้วย LogicalName ที่ได้จากขั้นตอนก่อน:

```sql
RESTORE DATABASE INVS
FROM DISK = '/var/opt/mssql/backup/INVS.bak'
WITH
  MOVE 'invs'     TO '/var/opt/mssql/data/INVS.mdf',
  MOVE 'invs_log' TO '/var/opt/mssql/data/INVS_log.ldf',
  REPLACE
```

<CodeExplainer title="อธิบาย RESTORE options">
  <details>
    <summary><code>MOVE ... TO ...</code></summary>
    <p>ระบุ path ใหม่บน Linux สำหรับแต่ละไฟล์</p>
  </details>
  <details>
    <summary><code>REPLACE</code></summary>
    <p>ถ้ามี database ชื่อ INVS อยู่แล้ว ให้ทับได้เลย</p>
  </details>
</CodeExplainer>

กระบวนการ restore ใช้เวลาขึ้นอยู่กับขนาดไฟล์ อาจใช้เวลาตั้งแต่ 30 วินาที ถึงหลายนาที ถ้าสำเร็จจะเห็น:

```
RESTORE DATABASE successfully processed 153842 pages in 47.823 seconds (25.143 MB/sec).
```

### 4.4 Troubleshooting — ปัญหาที่พบบ่อยตอน Restore

<InfoBox title="Cannot open backup device... Operating system error 5" type="danger">
  <p><strong>สาเหตุ:</strong> ไฟล์ยังไม่ได้ copy เข้า container หรือ copy ไปผิด path</p>
  <p><strong>แก้:</strong> ตรวจสอบอีกครั้ง:</p>

```bash
docker exec sqlserver ls -la /var/opt/mssql/backup/
```

</InfoBox>

<InfoBox title="The database was backed up on a server running version X..." type="warning">
  <p><strong>สาเหตุ:</strong> ไฟล์ backup ถูกสร้างจาก SQL Server เวอร์ชันใหม่กว่าที่รันอยู่</p>
  <p><strong>แก้:</strong> เปลี่ยน image เป็นเวอร์ชันที่ตรงกัน เช่น ถ้า backup มาจาก SQL Server 2019 ให้ใช้:</p>

```bash
docker run -d \
  --name sqlserver \
  -e "ACCEPT_EULA=Y" \
  -e "SA_PASSWORD=StrongPassword123!" \
  -p 1433:1433 \
  mcr.microsoft.com/mssql/server:2019-latest
```

</InfoBox>

<InfoBox title="Directory lookup for the file ... failed" type="warning">
  <p><strong>สาเหตุ:</strong> ไม่ได้ใส่ <code>MOVE</code> clause หรือใส่ LogicalName ผิด</p>
  <p><strong>แก้:</strong> รัน <code>RESTORE FILELISTONLY</code> อีกครั้งและตรวจสอบ LogicalName ให้ตรง</p>
</InfoBox>

## ส่วนที่ 6 — ติดตั้ง ODBC Driver บน macOS

### 6.1 ทำไมต้องใช้ ODBC Driver

<NarrativeSection>

Python ไม่สามารถคุยกับ SQL Server โดยตรงได้ ต้องผ่าน driver กลาง library `pyodbc` ทำหน้าที่เป็น bridge แต่ต้องการ driver ที่ติดตั้งใน OS ก่อน

</NarrativeSection>

### 6.2 ติดตั้งผ่าน Homebrew

```bash
brew tap microsoft/mssql-release https://github.com/Microsoft/homebrew-mssql-release
brew install msodbcsql17
brew install mssql-tools
```

ตรวจสอบว่าติดตั้งสำเร็จ:

```bash
odbcinst -q -d
```

ผลลัพธ์:

```
[ODBC Driver 17 for SQL Server]
```

<SideNote type="tip">
  <p><strong>ทำไมใช้ Driver 17 ไม่ใช่ 18?</strong> Driver 18 เปลี่ยน default ของ <code>Encrypt</code> จาก <code>No</code> เป็น <code>Yes</code> ทำให้ต้องเพิ่ม <code>TrustServerCertificate=yes</code> ใน connection string ซึ่งเราจะทำอยู่แล้ว แต่ Driver 17 มี compatibility กว้างกว่าและ error message ชัดเจนกว่าสำหรับ development</p>
</SideNote>

ถ้าต้องการใช้ Driver 18 แทน:

```bash
brew install msodbcsql18
```

แล้วเปลี่ยน `db_driver` ใน config เป็น `"ODBC Driver 18 for SQL Server"`

## ส่วนที่ 8 — สร้าง Config

### 8.1 สร้างไฟล์ .env

สร้างไฟล์ `.env` ที่ root ของ project:

```
DB_SERVER=localhost
DB_PORT=1433
DB_NAME=INVS
DB_USER=sa
DB_PASSWORD=StrongPassword123!
```

<InfoBox title="ความปลอดภัย" type="warning">
  <p>ไม่ควร commit ไฟล์ <code>.env</code> ขึ้น Git เพิ่ม <code>.env</code> ใน <code>.gitignore</code>:</p>

```bash
echo ".env" >> .gitignore
```

</InfoBox>

### 8.2 สร้างไฟล์ config.py

```python
from pydantic_settings import BaseSettings


class Settings(BaseSettings):
    # Database
    db_server: str = "localhost"
    db_port: int = 1433
    db_name: str = "INVS"
    db_user: str = "sa"
    db_password: str = ""
    db_driver: str = "ODBC Driver 17 for SQL Server"

    # Application
    app_host: str = "0.0.0.0"
    app_port: int = 8000

    class Config:
        env_file = ".env"
        case_sensitive = False


settings = Settings()
```

<NarrativeSection>

`pydantic-settings` จะอ่านค่าจาก environment variable ก่อน ถ้าไม่มีจึงใช้ค่า default ลำดับความสำคัญคือ: **Environment Variable → .env file → Default value**

</NarrativeSection>

## ส่วนที่ 9 — สร้าง Database Connection

### 9.1 สร้างไฟล์ database.py

```python
import pyodbc
from contextlib import contextmanager
from config import settings


def get_connection_string() -> str:
    """สร้าง connection string สำหรับ SQL Server"""
    return (
        f"DRIVER={{{settings.db_driver}}};"
        f"SERVER={settings.db_server},{settings.db_port};"
        f"DATABASE={settings.db_name};"
        f"UID={settings.db_user};"
        f"PWD={settings.db_password};"
        "TrustServerCertificate=yes;"
        "Encrypt=no;"
    )


@contextmanager
def get_db():
    """Context manager สำหรับ database connection"""
    conn = pyodbc.connect(get_connection_string())

    # แก้ปัญหา encoding ภาษาไทย
    conn.setdecoding(pyodbc.SQL_CHAR, encoding="utf-8")
    conn.setdecoding(pyodbc.SQL_WCHAR, encoding="utf-8")
    conn.setencoding("utf-8")

    try:
        yield conn
    finally:
        conn.close()
```

### 9.2 อธิบาย Connection String แต่ละ Parameter

<CodeExplainer title="Connection String Parameters">
  <details>
    <summary><code>DRIVER=&#123;ODBC Driver 17 for SQL Server&#125;</code></summary>
    <p>ชื่อ driver ที่ติดตั้งไว้ ต้องล้อมด้วย <code>&#123;&#125;</code> เป็น syntax ของ ODBC</p>
  </details>
  <details>
    <summary><code>SERVER=localhost,1433</code></summary>
    <p>format คือ <code>hostname,port</code> ใช้ comma ไม่ใช่ colon</p>
  </details>
  <details>
    <summary><code>TrustServerCertificate=yes</code></summary>
    <p>บอกให้ข้าม SSL certificate validation เพราะ SQL Server ที่รันใน Docker ใช้ self-signed certificate ถ้าไม่ใส่จะเจอ error "SSL Provider: The certificate chain was issued by an authority that is not trusted"</p>
  </details>
  <details>
    <summary><code>Encrypt=no</code></summary>
    <p>ปิด encryption สำหรับ development environment เพื่อความเร็ว production ควรเปิด</p>
  </details>
</CodeExplainer>

## ส่วนที่ 10 — ปัญหา Encoding ภาษาไทยและวิธีแก้

### 10.1 ที่มาของปัญหา

<NarrativeSection>

ฐานข้อมูล INVS ในโรงพยาบาลส่วนใหญ่ถูกสร้างบน Windows Server ซึ่งใช้ encoding ค่าเริ่มต้นเป็น `TIS-620` หรือ `Windows-874` (cp874) สำหรับภาษาไทย แต่ macOS ใช้ `UTF-8`

เมื่อ `pyodbc` อ่านข้อมูลจาก SQL Server โดยไม่ระบุ encoding จะพยายาม decode bytes ด้วย encoding ของระบบปฏิบัติการ ทำให้เกิด `UnicodeDecodeError` หรือข้อมูลภาษาไทยกลายเป็นตัวอักษรแปลก

</NarrativeSection>

### 10.2 วิธีแก้

```python
conn.setdecoding(pyodbc.SQL_CHAR, encoding="utf-8")
conn.setdecoding(pyodbc.SQL_WCHAR, encoding="utf-8")
conn.setencoding("utf-8")
```

<Table>
  <tr slot="head">
    <th>คำสั่ง</th>
    <th>ความหมาย</th>
  </tr>
  <Fragment slot="body">
    <tr>
      <td><code>setdecoding(SQL_CHAR, ...)</code></td>
      <td>กำหนด encoding สำหรับ column ประเภท CHAR, VARCHAR</td>
    </tr>
    <tr>
      <td><code>setdecoding(SQL_WCHAR, ...)</code></td>
      <td>กำหนด encoding สำหรับ column ประเภท NCHAR, NVARCHAR</td>
    </tr>
    <tr>
      <td><code>setencoding(...)</code></td>
      <td>กำหนด encoding สำหรับ parameter ที่ส่งไป SQL Server</td>
    </tr>
  </Fragment>
</Table>

<InfoBox title="รองรับทุก OS" type="success">
  <p>ข้อดีของวิธีนี้คือใช้ UTF-8 ซึ่งรองรับทุก OS ทั้ง macOS ✅, Windows ✅ และ Linux ✅</p>
</InfoBox>

### 10.3 Troubleshooting — ยังเจอปัญหา Encoding

ถ้ายังเจอ encoding ผิดหลังจากใช้ UTF-8 แล้ว ให้ตรวจสอบ collation ของฐานข้อมูล:

```sql
SELECT
    name,
    collation_name
FROM sys.databases
WHERE name = 'INVS'
```

<SideNote type="warning">
  <p>ถ้า collation เป็น <code>Thai_CI_AS</code> หรือ <code>Thai_BIN</code> ข้อมูลถูกเก็บในรูปแบบ Thai ซึ่งต้องการการ decode เพิ่มเติม ลอง: <code>conn.setdecoding(pyodbc.SQL_CHAR, encoding="tis-620")</code></p>
</SideNote>

## ส่วนที่ 11 — สร้าง FastAPI Application

### 11.1 สร้างไฟล์ main.py

```python
from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from database import get_db

app = FastAPI(
    title="INVS API",
    description="API สำหรับระบบคลังยา INVS",
    version="1.0.0",
)

# อนุญาตให้ frontend เรียก API ได้ (CORS)
app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],  # Vite default port
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)


@app.get("/health")
def health_check():
    """ตรวจสอบว่า API ทำงานอยู่"""
    return {"status": "ok"}


@app.get("/health/db")
def health_check_db():
    """ตรวจสอบว่าเชื่อมต่อฐานข้อมูลได้"""
    try:
        with get_db() as conn:
            cursor = conn.cursor()
            cursor.execute("SELECT 1")
            return {"status": "ok", "database": "connected"}
    except Exception as e:
        raise HTTPException(status_code=503, detail=str(e))


@app.get("/api/warehouses")
def get_warehouses():
    """ดึงข้อมูลคลังยาทั้งหมด"""
    with get_db() as conn:
        cursor = conn.cursor()
        cursor.execute("SELECT * FROM warehouse ORDER BY warehouse_id")
        cols = [c[0] for c in cursor.description]
        rows = cursor.fetchall()
        return [dict(zip(cols, row)) for row in rows]


if __name__ == "__main__":
    import uvicorn
    from config import settings

    uvicorn.run(
        "main:app",
        host=settings.app_host,
        port=settings.app_port,
        reload=True,
    )
```
